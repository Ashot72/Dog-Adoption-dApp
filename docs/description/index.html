Please read "How to run application" first.

The application is built with Nuxt.js which is the Vue.js framework https://nuxtjs.org/. Nuxt.js can run in three modes. First mode is Universal which

supports server-side rendering. The second one is Single Page Application mode (SPA). I built the app with SPA mode. Why it was not Universal app

as users can enjoy the fast speed provided by server-side rendering?

This application is dynamic. You can deploy the app on any network without touching the dApp code.

We use Web3 library to build Decentralized Applications. web3.js is a collection of libraries which allow us to interact with a local or remote Ethereum node,

using a HTTP or IPC connection. In other words, web3 is kind of like a portal into the Ethereum world. It is how we get programmatic access to the Ethereum

network. There is an issue with web3 and is that there are two groups of versions of web3 in the world right now.

First version is v0.x.x which uses callbacks for async code and another version is v1.x.x which supports promises and async/await.

With web3 just about everything we do is asynchronous in nature such as calling a function to a contract etc.

Web3 needs a provider which is communication layer. We have Metamask installed. Whenever Metamask is running inside of our browser

it automatically injects web3 library into any active page.

Provider is what stores accounts keys all the private keys and the public keys so we could not ignore it, but what we can do is to hijack the

provider that belongs to the injected version of web3

with no additional work required from us.

Figure 13

When we deploy contracts to Rinkeby test network we use web3 instance to facilitate our deployment process. We are going to create

Web3 instance and create a provider and that provider will tell our instance what network is supposed to be communicating with.

There is a big difference deploying to a test network and deploying to local network. The difference is that this time we have to make sure

that we are using a real account and we are going to spend some kind of fake but also some kind of real ether to do the deployment.

This time our provider needs to have an account that it is going to use as a source of ether for the deployment. To do that we are

going to use our account mnemonic which is a 12-word phrase that was provided by Metamask when we first created our account. Account mnemonic

stores not only your public key but also the private key and it can be used to unlock your account and use the ethers within it to deploy a contract

Truffle has done a lot of heavy lifting for us by allowing us to use the truffle-artifacts library which is itself a wrapper around the truffle-contract

which in turns wraps around web3.js library. Truffle HDWalletProvider is used to sign transactions for addresses derived from a 12-word mnemonic.

 

Another thing that we should consider is that we need to make sure that we have some node that we can connect to actually deploy our contract.

We have to connect to a node that exists inside the Rinkeby network. One of the common solutions is to run a local node on your own machine;

a real Ethereum node that connects to the Rinkeby network and you can use that node running on your machine to deploy your contract. However

that is a process that takes a lot of time and a lot of difficulty to set up. Instead, we can use a service called Infura https://infura.io/. Infura is a public API that we

can sign up for free. It gives us the ability to get access to a node that is hosted on the Rinkeby network. Infura Is also available on other test networks

and all the public networks involved with Ethereum.
